00022 /*
00023 Object Model implementing objects and containers.
00024 
00025 This module implements an abstraction for objects (with locks and
00026 reference counts), and containers for these user-defined objects,
00027 also supporting locking, reference counting and callbacks.
00028 
00029 The internal implementation of objects and containers is opaque to the user,
00030 so we can use different data structures as needs arise.
00031 */
00032 /*
00033 USAGE - OBJECTS
00034 
00035 An obj object is a block of memory that the user code can access,
00036 and for which the system keeps track (with a bit of help from the
00037 programmer) of the number of references around.  When an object has
00038 no more references (refcount == 0), it is destroyed, by first
00039 invoking whatever 'destructor' function the programmer specifies
00040 (it can be NULL if none is necessary), and then freeing the memory.
00041 This way objects can be shared without worrying who is in charge
00042 of freeing them.
00043 As an additional feature, obj objects are associated to individual
00044 locks.
00045 
00046 Creating an object requires the size of the object and
00047 and a pointer to the destructor function:
00048 
00049     struct foo *o;
00050 
00051     o = obj_alloc(sizeof(struct foo), my_destructor_fn);
00052 
00053 The value returned points to the user-visible portion of the objects
00054 (user-data), but is also used as an identifier for all object-related
00055 operations such as refcount and lock manipulations.
00056 
00057 On return from obj_alloc():
00058 
00059  - the object has a refcount = 1;
00060  - the memory for the object is allocated dynamically and zeroed;
00061  - we cannot realloc() the object itself;
00062  - we cannot call free(o) to dispose of the object. Rather, we
00063    tell the system that we do not need the reference anymore:
00064 
00065     obj_ref(o, -1)
00066 
00067   causing the destructor to be called (and then memory freed) when
00068   the refcount goes to 0.
00069 
00070 - obj_ref(o, +1) can be used to modify the refcount on the
00071   object in case we want to pass it around.
00072 
00073 - obj_lock(obj), obj_unlock(obj), obj_trylock(obj) can be used
00074   to manipulate the lock associated with the object.
00075 
00076 
00077 \section  USAGE - CONTAINERS
00078 
00079 An obj container is an abstract data structure where we can store
00080 obj objects, search them (hopefully in an efficient way), and iterate
00081 or apply a callback function to them. A container is just an obj object
00082 itself.
00083 
00084 A container must first be allocated, specifying the initial
00085 parameters. At the moment, this is done as follows:
00086 
00087     <b>Sample Usage:</b>
00088     \code
00089 
00090     struct obj_container *c;
00091 
00092     c = obj_container_alloc(MAX_BUCKETS, my_hash_fn, my_cmp_fn);
00093     \endcode
00094 
00095 where
00096 
00097 - MAX_BUCKETS is the number of buckets in the hash table,
00098 - my_hash_fn() is the (user-supplied) function that returns a
00099   hash key for the object (further reduced modulo MAX_BUCKETS
00100   by the container's code);
00101 - my_cmp_fn() is the default comparison function used when doing
00102   searches on the container,
00103 
00104 A container knows little or nothing about the objects it stores,
00105 other than the fact that they have been created by obj_alloc().
00106 All knowledge of the (user-defined) internals of the objects
00107 is left to the (user-supplied) functions passed as arguments
00108 to obj_container_alloc().
00109 
00110 If we want to insert an object in a container, we should
00111 initialize its fields -- especially, those used by my_hash_fn() --
00112 to compute the bucket to use.
00113 Once done, we can link an object to a container with
00114 
00115    obj_link(c, o);
00116 
00117 The function returns NULL in case of errors (and the object
00118 is not inserted in the container). Other values mean success
00119 (we are not supposed to use the value as a pointer to anything).
00120 Linking an object to a container increases its refcount by 1
00121 automatically.
00122 
00123 \note While an object o is in a container, we expect that
00124 my_hash_fn(o) will always return the same value. The function
00125 does not lock the object to be computed, so modifications of
00126 those fields that affect the computation of the hash should
00127 be done by extracting the object from the container, and
00128 reinserting it after the change (this is not terribly expensive).
00129 
00130 \note A container with a single buckets is effectively a linked
00131 list. However there is no ordering among elements.
00132 */
